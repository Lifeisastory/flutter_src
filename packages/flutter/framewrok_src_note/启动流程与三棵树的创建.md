# Flutter 启动流程与三棵树创建实现分析

## 1. 分析范围

- 本次只分析 Flutter framework 在 `runApp` 启动阶段，如何从根 widget 启动，到首帧前后完成 Widget Tree、Element Tree、Render Tree 的创建与接线。
- 本次重点覆盖的三棵树是:
  - `Widget Tree`
  - `Element Tree`
  - `Render Tree`
- 本次不展开:
  - Material / Cupertino 业务组件细节
  - 后续普通 rebuild 流程
  - 语义树的完整生命周期
  - layer tree / GPU 合成细节，只在与首帧创建直接相关时顺带提及
- 项目总体索引入口: [ARCHITECTURE_INDEX.en.md](packages/flutter/framewrok_src_note/ARCHITECTURE_INDEX.en.md)

## 2. 功能概览

- 对外表现上，Flutter 启动看起来只是调用了一次 `runApp(app)`，但内部实际上分成两段:
  - 先建立 framework 级的根 widget / 根 element
  - 再在默认 `View` 内部创建独立 render tree，并把它接入 `RendererBinding`
- 核心流程摘要:
  1. `runApp` 确保 `WidgetsFlutterBinding` 已初始化。
  2. binding 用默认 `FlutterView` 把用户传入的 app 包成 `View`。
  3. binding 把这个根 widget 包成 `RootWidget`，并挂到 `BuildOwner` 上，形成 root element。
  4. root element 在 build 中一路向下 inflate，到 `View -> RawView -> _RawViewInternal`。
  5. `_RawViewInternal` 对应的 `_RawViewElement` 在 mount 时创建 `RenderView` 根节点，并把它挂到新的 `PipelineOwner`。
  6. 首帧 `drawFrame` 时先跑 widget build，再跑 render pipeline，于是三棵树真正联动起来。

## 4. 主流程详解

### 4.1 起点

- `runApp` 先执行 `WidgetsFlutterBinding.ensureInitialized()`，确保 framework 顶层 binding 已存在，然后调用 `_runWidget(binding.wrapWithDefaultView(app), ...)`。证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1614-L1617)
- `WidgetsBinding.initInstances` 里会创建 `BuildOwner`，并把 `onBuildScheduled` 绑定到 framework 自己的调度逻辑。证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L463-L476)
- `wrapWithDefaultView` 不会直接创建 render tree，而是把用户传入的 `rootWidget` 包成一个 `View(view: implicitView, child: rootWidget)`。这一步的意义是:
  - 给 app 指定默认 `FlutterView`
  - 为后面在 `View` 内部 bootstrap render tree 做准备
    证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1350-L1380)

### 4.2 中间调度与根节点挂载

- `_runWidget` 做了两件事:
  - `scheduleAttachRootWidget(app)`
  - `scheduleWarmUpFrame()`
    也就是说，Flutter 启动不是同步一把梭到底，而是“先安排根挂载，再安排 warm-up frame”。证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1679-L1683)
- `scheduleAttachRootWidget` 使用 `Timer.run` 异步调用 `attachRootWidget(rootWidget)`。证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1383-L1391)
- `attachRootWidget` 会把传入 widget 再包一层 `RootWidget(debugShortDescription: '[root]', child: rootWidget)`，然后交给 `attachToBuildOwner`。证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1394-L1405)
- `attachToBuildOwner` 是 framework 根挂载点:
  - 如果这是第一次启动，`rootElement == null`
  - 设置 `_readyToProduceFrames = true`
  - 调用 `widget.attach(buildOwner!, rootElement as RootElement?)`
  - 首次启动时调用 `SchedulerBinding.instance.ensureVisualUpdate()` 请求可视更新
    证据: [packages/flutter/lib/src/widgets/binding.dart](../lib/src/widgets/binding.dart#L1416-L1422)

### 4.3 Element Tree 是怎么创建出来的

- `RootWidget.attach` 是 element tree 真正的启动器。
- 首次启动时它会:
  - `element = createElement()` 创建 `RootElement`
  - `element.assignOwner(owner)` 把 `BuildOwner` 赋给根
  - `owner.buildScope(element!, () { element!.mount(null, null); })`
    证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1736-L1750)
- `RootElementMixin.assignOwner` 并不只是塞一个字段，它同时:
  - 设置 `_owner = owner`
  - 建立根级别的 `BuildScope`
    证据: [packages/flutter/lib/src/widgets/framework.dart](packages/flutter/lib/src/widgets/framework.dart#L7028-L7042)
- 后续整棵 element tree 通过通用路径逐层展开:
  - 父 element 调 `updateChild`
  - 若没有旧 child，则走 `inflateWidget`
  - `inflateWidget` 里调用 `newWidget.createElement()`
  - 然后 `newChild.mount(this, newSlot)`
    证据:
  - [packages/flutter/lib/src/widgets/framework.dart](packages/flutter/lib/src/widgets/framework.dart#L3982-L4059)
  - [packages/flutter/lib/src/widgets/framework.dart](packages/flutter/lib/src/widgets/framework.dart#L4556-L4589)
- 也就是说:
  - `Widget Tree` 是配置树，来自代码里的 widget 对象
  - `Element Tree` 是运行期实例树，是在 `createElement + mount` 阶段逐层生成的

### 4.4 Render Tree 是怎么创建出来的

- 根 element 往下 build 到 `View` 后，`View.build` 会返回 `RawView(...)`。证据: [packages/flutter/lib/src/widgets/view.dart](packages/flutter/lib/src/widgets/view.dart#L248-L259)
- `RawView.build` 再返回 `_RawViewInternal(...)`。证据: [packages/flutter/lib/src/widgets/view.dart](packages/flutter/lib/src/widgets/view.dart#L370-L382)
- `_RawViewInternal` 是一个 `RenderObjectWidget`，它有两个关键点:
  - `createElement() => _RawViewElement(this)`
  - `createRenderObject(...) => RenderView(view: view)`
    证据: [packages/flutter/lib/src/widgets/view.dart](packages/flutter/lib/src/widgets/view.dart#L407-L443)
- 这意味着 render tree 的根并不是在 `runApp` 里直接 new 出来的，而是在 `_RawViewInternal` 被 inflate 成 `_RawViewElement` 后，由 `RenderObjectElement.mount` 和 `_RawViewElement.mount` 联手建立。

## 5. 关键实现细节

- 普通 `Element.mount` 负责把 element 接入 element tree:
  - 记录 `_parent`
  - 设置 `_slot`
  - 进入 active 生命周期
  - 从父节点继承 `owner`
  - 处理 `GlobalKey`
    证据: [packages/flutter/lib/src/widgets/framework.dart](packages/flutter/lib/src/widgets/framework.dart#L4328-L4360)
- `RenderObjectElement.mount` 比普通 element 多做了两件决定性的事:
  - `_renderObject = (widget as RenderObjectWidget).createRenderObject(this)`
  - `attachRenderObject(newSlot)`
    证据: [packages/flutter/lib/src/widgets/framework.dart](packages/flutter/lib/src/widgets/framework.dart#L6784-L6802)
- 对 `_RawViewElement` 来说，render tree 的根级创建逻辑集中在 `mount`:
  - `_effectivePipelineOwner.rootNode = renderObject`
  - `_attachView()`
  - `_updateChild()`
  - `renderObject.prepareInitialFrame()`
    证据: [packages/flutter/lib/src/widgets/view.dart](packages/flutter/lib/src/widgets/view.dart#L499-L508)
- `_attachView()` 做了 render tree 参与全局渲染循环所必须的两件事:
  - `parentPipelineOwner.adoptChild(_effectivePipelineOwner)`
  - `RendererBinding.instance.addRenderView(renderObject)`
    证据: [packages/flutter/lib/src/widgets/view.dart](packages/flutter/lib/src/widgets/view.dart#L513-L518)
- `RenderView.prepareInitialFrame()` 则做首帧前的 render pipeline bootstrap:
  - `scheduleInitialLayout()`
  - `scheduleInitialPaint(_updateMatricesAndCreateNewRootLayer())`
    证据: [packages/flutter/lib/src/rendering/view.dart](packages/flutter/lib/src/rendering/view.dart#L235-L261)
- 这里可以看出一个很容易忽略的设计点:
  - `RenderView` 创建出来不等于 render tree 已经“跑起来”
  - 真正让它进入首帧工作状态的是: `rootNode` 赋值 + `addRenderView` 注册 + `prepareInitialFrame`

## 7. 时序总结

1. 用户调用 `runApp(app)`。
2. `WidgetsFlutterBinding.ensureInitialized()` 创建并初始化 binding。
3. `WidgetsBinding.initInstances` 创建 `BuildOwner`。
4. `wrapWithDefaultView` 把 `app` 包成默认 `View`。
5. `_runWidget` 调度 `attachRootWidget` 和 `scheduleWarmUpFrame`。
6. `attachRootWidget` 把根再包成 `RootWidget`。
7. `RootWidget.attach` 创建 `RootElement`，并把 `BuildOwner` 赋给根。
8. `RootElement.mount` 开始递归 inflate 子 widget，逐步生成 element tree。
9. 递归 build 到 `View -> RawView -> _RawViewInternal`。
10. `_RawViewInternal` 作为 `RenderObjectWidget` 被 mount 时创建 `RenderView`。
11. `_RawViewElement.mount` 把 `RenderView` 设为 `PipelineOwner.rootNode`，再注册到 `RendererBinding`。
12. `RenderView.prepareInitialFrame` 安排初始 layout / paint。
13. 首帧 `WidgetsBinding.drawFrame` 先执行 `buildOwner.buildScope(rootElement!)`。
14. 然后 `RendererBinding.drawFrame` 执行 `flushLayout -> flushCompositingBits -> flushPaint -> compositeFrame`。

## 8. 边界情况与注意点

- 这三棵树不是同一个时间点“一次性同时创建”的:
  - widget tree 在代码层面先存在
  - element tree 在 `attach/mount/inflateWidget` 中逐层实例化
  - render tree 在遇到 `RenderObjectWidget`，尤其 `_RawViewInternal` 时才真正开始创建
- `runApp` 当前默认依赖 `platformDispatcher.implicitView`；如果没有默认 view，会抛错并提示改用 `runWidget`。证据: [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart#L1359-L1373)
- `RootWidget` / `RootElement` 是 framework 根，但不直接等于 render tree 根。
- render tree 根是 `RenderView`，其创建点在 [packages/flutter/lib/src/widgets/view.dart](packages/flutter/lib/src/widgets/view.dart)，不是 [packages/flutter/lib/src/widgets/binding.dart](packages/flutter/lib/src/widgets/binding.dart)。
- 推断: 你如果把“三棵树”理解成 widget / element / render，是和当前源码实现最吻合的；如果把第三棵树理解成 layer tree，则需要继续下钻 `RenderView.compositeFrame` 与 layer 建树逻辑，本次未展开。
